Little Fig Documentation

Little Fig is a training and inference engine for large language models designed around one principle: fit in less RAM. Standard LLM fine-tuning requires 16-32× the model's parameter count in memory. A 1.1B model needs ~27GB in FP32. Fig Engine brings that to ~400MB by combining adaptive INT4 quantization, fused compute kernels, and four training strategies that each optimize for different memory budgets.

The engine also integrates Ember's Diaries — a cognitive memory database — directly into the training pipeline, allowing you to teach a model to store, recall, consolidate, and forget information using special tokens embedded in its vocabulary.

FigQuant — Adaptive Codebook Quantization

FigQuant compresses model weights from 32-bit floats to 4-bit integers using a learned codebook. Unlike standard INT4 (which uses fixed min/max scaling) or NF4 (which assumes weights are normally distributed), FigQuant adapts its codebook to the actual weight distribution of each layer.

How it works
  1. Group weights into blocks of 128 elements and compute per-group absmax scaling
  2. Initialize codebook from NF4 quantiles (16 values representing a normal distribution)
  3. Refine via weighted k-means — high-magnitude weights receive more importance (AWQ-inspired sensitivity weighting), pushing the codebook to allocate more precision where it matters most
  4. Assign + pack — each weight maps to the nearest codebook entry (4 bits), two per byte
  5. Double quantization — the per-group scale factors themselves are quantized to FP8, saving ~0.37 bits/param

FigKernel — Fused Operations

FigKernel provides fused operations that reduce memory traffic and speed up computation. Unlike Triton kernels (GPU-only), these are written in standard PyTorch and compiled by torch.compile(backend="inductor") — generating optimized AVX-512 code on CPU and CUDA code on GPU from the same source.

FigRMSNorm

Fuses variance computation, rsqrt, and scale into one pass. Saves only inv_rms per row for backward (scalar) instead of the full normalized tensor. 2.95× faster than standard PyTorch RMSNorm. Auto-swapped into models at load time.

FigCrossEntropy

Chunked cross-entropy that never materializes the full [seq_len, vocab_size] logit tensor. For a 128K vocabulary, this avoids a 512KB allocation per token. Processes the vocabulary in 8K chunks using numerically stable running logsumexp. ~8× less peak memory.

FigSwiGLU

Fused gate + up + SiLU activation. The inductor backend merges the two linear projections and activation into a single vectorized pass, saving one full hidden-state-sized activation per layer.

fig_fused_linear_lora

The critical path: F.linear(x, W) + (x @ A) @ B * scale compiled into one kernel. Every FigLinear forward pass uses this when cached weights are available. Zero overhead from LoRA on the fused path.

FigPipeline — GPU-CPU Async Training

When a GPU is available, FigPipeline keeps optimizer states (AdamW's exp_avg and exp_avg_sq) on CPU while running forward and backward passes on GPU. This enables training models that exceed GPU VRAM.

Architecture

Small parameters (<100K elements) — updated directly on GPU. The overhead of transferring to CPU exceeds the cost of running AdamW on GPU. LoRA adapters (rank 16 × hidden_dim) always take this path.

Large parameters (>100K elements) — gradients transferred to CPU via async CUDA streams, AdamW update runs on CPU, updated weights sent back. Overlapped with the next layer's forward computation.

For LoRA training (rank 16), gradient transfer per layer is ~100KB — negligible vs PCIe bandwidth. FigPipeline is auto-enabled when GPU is detected and use_pipeline=True in training config.

Training Tiers

Fig Engine automatically selects the best training strategy based on available RAM. The system budgets 70% of free memory (30% headroom for OS) and picks the highest-quality tier that fits.

Tier 1Streaming LoRA

FigQuant INT4 base weights + FP32 low-rank adapters. Only the LoRA matrices (A and B) are trainable — typically 1-3% of total parameters. Base weights dequantize on-the-fly during forward/backward. ~400MB for a 1.1B model. The default and fastest tier.

Reference: Hu et al., "LoRA: Low-Rank Adaptation of Large Language Models" (ICLR 2022)

Tier 2LISA

Layerwise Importance Sampled AdamW. Observes that LoRA disproportionately updates embedding and head layers (100× more than middle layers). LISA freezes all but γ randomly sampled middle layers every K steps, achieving 10-35% better quality than LoRA. ~900MB for 1.1B.

Reference: Pan et al., arxiv 2403.17919 (2024)

Tier 3MeZO

Memory-efficient Zeroth-Order optimization. Eliminates backpropagation entirely — estimates gradients via two forward passes with random perturbation. Memory equals inference cost. Acceptable quality on most tasks, but slow (2× forward passes per step). ~600MB for 1.1B.

Reference: Malladi et al., "Fine-Tuning LLMs with Just Forward Passes" (NeurIPS 2023)

Tier 4LOMO

Low-Memory Optimization. Fuses gradient computation with parameter updates during backpropagation — maintains only one gradient tensor at any time (O(1) gradient memory). Enables full-parameter training at near-LoRA memory cost. Best quality, ~800MB for 1.1B.

Reference: Lv et al., arxiv 2306.09782 (2023)

Memory Optimizations

Beyond the training tiers, Fig Engine provides three additional memory optimizations that can be combined with any tier.

Activation Checkpointing

Recomputes intermediate activations during the backward pass instead of storing them. Saves ~70-80% of activation memory at the cost of ~33% extra compute. Enabled by default via activation_checkpointing=True. Uses HuggingFace's native gradient checkpointing when available, falls back to torch.utils.checkpoint with use_reentrant=False.

Adafactor Optimizer

Replaces AdamW with Adafactor, which factorizes the second moment matrix into row and column vectors. For a 1024×1024 matrix, this reduces optimizer state from ~2M elements to ~2K elements — 1000× reduction per matrix. Enable via use_adafactor=True. Quality is comparable to AdamW on most tasks.

Reference: Shazeer & Stern, "Adafactor: Adaptive Learning Rates with Sublinear Memory Cost" (ICML 2018)

CPU BF16 Autocast

Runs forward and backward passes in bfloat16 on CPU. BF16 has the same dynamic range as FP32 (8 exponent bits) so no gradient scaler is needed, unlike FP16. Halves activation memory during compute. Enable via cpu_bf16=True. Gradients accumulate in FP32 automatically.

Ember Memory Integration

Ember's Diaries is a cognitive database engine built on the principle that nothing is ever deleted. Little Fig integrates Ember by injecting 9 special tokens into the model's vocabulary and training it to perform memory operations as part of its text generation.

Memory Tokens
<|mem_store|>
Persist information to memory
<|mem_recall|>
Search and retrieve from memory
<|mem_consolidate|>
Merge related memories (sensory→long-term)
<|mem_forget|>
Flag decayed memories (Ebbinghaus curves)
<|mem_conflict|>
Detect and track contradictions
<|mem_episode|>
Mark conversation episode boundaries
<|mem_reflect|>
Metacognitive self-reflection
<|memory_start|>
Begin memory operation block
<|memory_end|>
End memory operation block

When ember_mode=True, the model's tokenizer is extended with these tokens and its embedding layer is resized. The load_ember_dataset() method generates synthetic training examples where the model must decide when to store, what to recall, how to consolidate, and when to flag decayed or conflicting memories.

The trained model can then operate its own Ember's Diaries instance — it's not RAG (external retrieval), it's cognitive memory built into the model's weights.

Universal GGUF Loader

Little Fig can load any .gguf file automatically — no hardcoded architecture mappings. Three loading strategies are tried in order:

  1. transformers built-in — covers 22+ architectures natively
  2. gguf-py TensorNameMap — falls back to gguf-py's knowledge of 122+ architectures
  3. Heuristic direct loading — downloads HF config from Hub, builds GGUF→HF name map by convention

Python API Reference

FigModel.from_pretrained(model_name, ...)

Load a HuggingFace model with FigQuant INT4 + LoRA adapters. Key params: lora_r, lora_alpha, tier, ember_mode, fuse_kernels, fast (cached vs on-the-fly dequant), compile_model.

FigTrainer(model, config)

Unified trainer. Methods: load_dataset(source), load_ember_dataset(n_examples), train(). Auto-selects tier and uses FigPipeline when GPU available.

FigTrainingConfig

Training configuration. Key params: tier, num_epochs, learning_rate, max_seq_length, activation_checkpointing, use_adafactor, cpu_bf16, use_pipeline, push_to_hub.

FigModel.save_adapter(path) / load_adapter(path)

Save/load only LoRA adapter weights (~5MB). The adapter file is portable — share it without the full base model.

FigModel.merge_and_export(path)

Merge LoRA adapters into base weights and save as a standard HuggingFace model. Ready for inference or push to Hub.

FigModel.push_to_hub(repo_id)

Push trained model to HuggingFace Hub. Can push merged model or adapter-only.
